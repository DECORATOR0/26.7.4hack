# InternCast Latest Integration Notes

日期：2026-07-05

来源包：`interncast_latest_20260705_024931.zip`

## 核心判断

对方版本最值得吸收的部分不是整体事件识别，而是进球判定逻辑更硬、更工程化。

当前我们这版仍然比较依赖视觉叙述和模型综合判断：看到球入网、庆祝、进球信息条、回放画面时，模型容易把同一次进球的后续画面误判成新进球。对方版本把进球从普通视觉事件里拆出来，改成以记分牌 OCR 的比分跳变作为硬标准。

## 对方进球方案

对方流程大致是：

1. 固定裁剪转播画面右上角记分牌区域。
2. 批量调用 Intern-S2 OCR，读取 `home_score`、`away_score`、`clock`、`visible`、`confidence`、`replay`。
3. 只把比分从旧值到新值且总进球数正好增加 1 的位置判定为进球。
4. 如果粗扫两帧之间发现比分变化，再对该窗口做 1 秒级细扫，尽量定位第一个比分跳变点。
5. OCR 标记为 `replay=true`、低置信度、比分回退、比分一次跳多球的读数都不作为进球依据。

这套方案本质上不依赖球员身份、射门动作、球是否实际入网等语义判断，而是把“新进球”绑定到“直播记分牌真的变了”。这正好解决 v4.2 里假进球偏多的问题。

## 非进球事件

角球、任意球、犯规、越位、换人、争议等其他事件，对方思路和我们目前版本接近，仍然是抽帧后用视觉模型识别，再通过规则做筛选和合并。

可借鉴的筛选规则：

- 普通回放不作为独立事件。
- 视觉-only 的 `goal_or_celebration` 没有比分跳变支持时，不进入最终进球列表。
- 同一分钟内的相关事件可以合并成一个讲解节点。
- 犯规和随后任意球可以合并。
- 角球区证据明确时优先判角球，不判模糊任意球。
- 点球必须有点球点、点球准备、裁判明确指点球点等强证据，否则降级为犯规/争议。
- 换人必须看到换人牌或明确换人字幕，只看到场边人员不算。

## 最终输出风格

对方导出的 `fused_commentary.md` 风格可以直接参考：

- 结构更完整，分成解说脚本、关键事件时间轴、多语言解说、配音脚本、字幕脚本、集锦讲解脚本、技术统计、内部复核备注。
- 文案面向交付，不暴露 `pending`、`needs_review`、密采窗口等工程词。
- 不输出具体球员姓名，尽量使用球队级描述。
- 事件表使用分钟级表达，文案按事件组推进，不把回放/庆祝重复写成新事件。
- 技术统计需要明确说明只基于 Harness 确认事件，不等同官方统计。

后续最终报告可以直接抄这版输出框架，但事件来源必须换成我们校验后的事件表。

## 建议落地顺序

第一步先改进球判定：

- 新增记分牌裁剪和 OCR reading 数据结构。
- 新增比分跳变检测。
- 新增粗扫后的 1 秒级细扫。
- final events 中的 `goal` 必须有比分跳变证据。
- 视觉模型识别出的进球/庆祝只作为辅助证据或候选回放，不直接生成新进球。

第二步再改最终输出：

- 保留现有事件检测流程。
- 在 script report 阶段改成对方的 fused commentary 风格。
- 把事件表、讲解脚本、字幕、口播、集锦说明统一成一份 Markdown。

第三步补回归样例：

- 回放不产生新进球。
- 进球信息条不产生新进球。
- 比分未变时不产生新进球。
- 视觉-only 进球无比分支持时过滤。
- 点球和点球进球不重复计数。
- 角球/任意球/犯规相邻事件按规则合并。

## 风险

对方 zip 自带示例输出不代表整套流程已经高准确率跑通，示例里有 API 连接失败 warning，也还有普通进攻、换人、视觉进球等误判。可直接借鉴的是设计和规则，不建议直接替换整个 pipeline。

另外，记分牌裁剪区域依赖转播样式。如果换视频源或比分牌位置变化，需要把 crop 区域做成配置或自动探测。
