# 视觉事件识别方案讨论稿

## 1. 目标

本方案只考虑视觉模态，不依赖音频 ASR。

输入：

```text
match.mp4
```

输出：

```text
candidate_segments.json
visual_events.json
events.json
```

目标不是逐帧精确理解整场比赛，而是从长视频中自动找出适合解说的关键事件候选，再让 Intern-S2 对少量候选片段做分类和描述。

## 2. 为什么不能全量把图片塞给模型

一场 110 分钟视频：

```text
110 分钟 * 60 秒 / 5 秒 = 1320 张图
```

即使每张图压缩到很小，也不适合全部送入 256K 上下文。

原因：

- 图片 token 成本高。
- 大量普通比赛画面信息密度低。
- 模型看太多图容易遗漏真正关键事件。
- 请求时间会变长，现场演示不稳定。

所以必须做两级流程：

```text
全视频便宜处理 -> 筛候选片段 -> 只把少量候选片段交给 Intern-S2
```

## 3. 抽帧成本

5 秒抽一帧本身不会特别慢。

对 110 分钟视频，大约 1320 帧。用 OpenCV 抽 JPEG，通常是分钟级以内，具体取决于机器和视频编码。当前已有代码用很稀疏参数抽过 7 帧，几秒内完成。抽到 1000 多帧会更慢，但仍然比让模型看 1000 张图便宜得多。

建议：

```text
开发调试：每 30-60 秒抽一帧
正式候选筛选：每 5 秒抽一帧
候选片段密抽：每 1 秒或 0.5 秒抽一帧
```

不要把 5 秒抽帧结果全部给模型，它只是本地筛候选用。

## 4. 亮度分数是否有用

单帧亮度本身用处不大。

```text
brightness(frame)
```

只能知道画面明暗，不能判断是否有关键事件。

真正有用的是帧间变化：

```text
diff(frame_t, frame_t-1)
```

也就是当前帧和上一帧在画面上的差异。它可以提示：

- 画面切换
- 回放转场
- 近景切换
- 大面积运动
- 镜头从比赛全景切到球员庆祝或裁判特写

所以候选筛选不要看静态亮度，而要看：

```text
motion_score
scene_change_score
overlay_change_score
scoreboard_change_score
```

## 5. 候选片段怎么筛

第一层不要识别具体事件，只判断“这里可能有事”。

建议每 5 秒抽一帧，计算这些信号：

### 5.1 帧间变化

```text
motion_score = 当前帧和上一帧的像素差异
```

高分可能表示：

- 镜头切换
- 快速推进
- 慢动作回放转场
- 庆祝近景

但它不是事件本身，只能作为候选信号。

### 5.2 镜头类型变化

可以用简单规则估计：

```text
wide_field_view      大范围绿色草坪
close_up             人物近景
crowd_view           看台/观众
scoreboard_overlay   转播比分条明显
```

关键事件前后经常出现：

```text
比赛全景 -> 球员近景 -> 庆祝/回放 -> 比分牌
```

### 5.3 转播图层变化

如果画面上出现新的大面积字幕条、比分条、回放条，可能意味着：

- 进球后比分更新
- 换人
- 黄牌/红牌
- VAR/判罚说明
- 回放

第一版可以先不做复杂 OCR，只检测画面上下区域变化：

```text
top_overlay_diff
bottom_overlay_diff
```

### 5.4 草坪比例变化

足球比赛常规画面通常大量绿色草坪。

如果绿色比例突然变低，可能表示：

- 球员近景
- 裁判特写
- 替补席
- 看台庆祝
- 回放特写

这个信号对候选筛选有用。

### 5.5 候选帧合并成候选片段

不要单独处理某一帧。

如果某一帧高分，取它前后窗口：

```text
candidate_center = 00:31:20
segment_start = center - 15s
segment_end = center + 20s
```

多个相近候选合并：

```text
00:31:20, 00:31:25, 00:31:30
-> 00:31:05 - 00:31:50
```

最终输出：

```json
{
  "segment_id": "C012",
  "start": "00:31:05",
  "end": "00:31:50",
  "center": "00:31:25",
  "signals": ["scene_change", "close_up", "overlay_change"],
  "candidate_score": 0.82
}
```

## 6. 候选片段给模型怎么输入

不能给全视频，也不要给 1320 张粗抽帧。

只给候选片段，每个片段 4-8 张图：

```text
start
center - 5s
center
center + 5s
end
```

如果片段较复杂，可以给 8 张：

```text
每 5 秒 1 张，最多 8 张
```

一次请求不要塞太多候选片段。

建议批处理：

```text
每次请求处理 3-5 个候选片段
每个候选片段 4-6 张图
总图片数控制在 20-30 张以内
```

这样即使 256K 上下文足够，也能降低模型遗漏和延迟。

## 7. 统一分类 vs 每类单独识别

### 7.1 统一分类器

做法：

```text
给一组候选片段 + 固定事件标签定义
让模型从标签里选
```

标签：

```text
goal_or_celebration
replay
corner
free_kick
penalty
card
substitution
referee_dispute
attack_highlight
unknown
```

优点：

- 实现快。
- prompt 统一。
- 适合第一版跑通。

缺点：

- 标准太多时，模型可能混淆。
- 对黄牌、换人、角球这类细节可能漏判。

### 7.2 分事件识别器

做法：

```text
先用统一分类器粗分
再对高价值候选调用专门识别器
```

例如：

```text
CardDetector
SubstitutionDetector
SetPieceDetector
ReplayCelebrationDetector
```

优点：

- 每个 detector 标准更清楚。
- 对特定事件更稳。

缺点：

- 请求次数多。
- 工程复杂度高。
- 黑客松时间压力下容易做不完。

### 7.3 推荐

第一版：

```text
统一 EventClassifier
```

第二版：

```text
统一 EventClassifier + 少量专门 verifier
```

例如：

```text
如果 EventClassifier 判断为 card/substitution/penalty，
再调用对应 verifier 确认一次。
```

这比一开始为每个事件写独立流程更稳。

## 8. 各事件视觉标准

### 8.1 goal_or_celebration

好抓程度：高。

视觉信号：

- 多名队友拥抱、奔跑、滑跪。
- 进球方球员朝看台或角旗区庆祝。
- 对方球员低头、摊手、门将回头看球网。
- 转播画面切到近景或看台。
- 比分牌可能变化。

注意：

- 单独庆祝可以判为 `goal_or_celebration`。
- 是否真进球最好结合比分牌变化验证。

### 8.2 replay

好抓程度：高。

视觉信号：

- 画面出现 Replay 标识或转播回放特效。
- 慢动作特写。
- 同一动作多角度重复。
- 画面色调/速度明显不同。

注意：

- 回放通常跟在进球、犯规、争议判罚或精彩机会之后。
- 回放本身不是事件，但可以强化前一个事件的可信度。

### 8.3 corner

好抓程度：中高。

视觉信号：

- 球员站在角旗附近。
- 角旗、底线、边线同时出现。
- 禁区内多人拥挤等待争顶。
- 镜头固定在角旗区。

风险：

- 边线球、角旗附近控球可能误判为角球。

### 8.4 free_kick

好抓程度：中。

视觉信号：

- 定点球。
- 人墙站成一排。
- 裁判指挥位置。
- 禁区前多人站位。

风险：

- 普通定位球、开球、犯规后暂停都可能相似。

### 8.5 penalty

好抓程度：中高。

视觉信号：

- 球摆在点球点。
- 门将站在门线。
- 其他球员站在禁区外。
- 镜头正对球门和主罚球员。

风险：

- 正常单刀前的画面可能被误判，需要多帧确认。

### 8.6 card

好抓程度：中高。

视觉信号：

- 裁判近景。
- 裁判手举黄牌或红牌。
- 球员围住裁判。
- 转播字幕显示犯规/球员信息。

风险：

- 单张图里牌很小，模型可能看不清。
- 需要候选片段里至少有裁判特写。

### 8.7 substitution

好抓程度：中高。

视觉信号：

- 场边第四官员举牌。
- 两名球员在边线交接。
- 替补席画面。
- 转播字幕显示换人。

风险：

- 普通边线画面可能误判。

### 8.8 referee_dispute

好抓程度：中。

视觉信号：

- 多名球员围住裁判。
- 球员摊手、抗议。
- 裁判解释手势。
- VAR 标识或裁判指向耳机/屏幕。

风险：

- 普通犯规后的交涉和争议判罚很像。

### 8.9 attack_highlight

好抓程度：中。

这是兜底类别。

视觉信号：

- 禁区内多人密集。
- 球门附近高强度攻防。
- 门将倒地、后卫封堵、球员射门姿态。

注意：

- 不要细分为 `save`、`shot_on_target`、`woodwork`，除非证据非常明确。
- 解说时只能说“形成威胁攻势”，不要说“击中门框”或“神扑”。

## 9. 球是否需要识别

第一版不建议依赖“球的轨迹识别”。

原因：

- 足球在转播画面里很小。
- 压缩后容易模糊。
- 高速运动时单帧很难找到。
- 球的位置不等于事件类型。

可以做简单辅助：

```text
如果能检测到球靠近球门/角旗区，可提高候选分数。
```

但不要把它作为主判断依据。

更稳的信号是：

```text
转播镜头变化
球员群体行为
裁判/场边/回放特写
比分牌或字幕图层变化
```

## 10. 上下文和图片预算

建议预算：

```text
粗抽帧：本地 1000-1500 张，不进模型
候选片段：Top 20-40 个
每个片段：4-6 张图
进入模型：分批处理，每批 3-5 个片段
每批图片：20-30 张以内
```

模型输入应该是：

```text
候选片段元数据 + 少量代表帧 + 固定标签定义
```

不是：

```text
全量帧 + 让模型自己找事件
```

## 11. 第一版落地建议

第一版实现顺序：

1. 每 5 秒抽一帧，生成 `frame_index.json`。
2. 计算帧间变化、草坪比例、上下区域 overlay 变化。
3. 选 Top 30 候选中心帧。
4. 相邻候选合并成候选片段。
5. 每个片段密抽 4-6 张图。
6. 用统一 `EventClassifier` 调 Intern-S2 分类。
7. 合并相邻重复事件。
8. 输出 `visual_events.json`。
9. 再根据 `visual_events.json` 生成 `events.json` 和解说。

## 12. 关键原则

- 不全量喂图。
- 不依赖球轨迹。
- 第一版不细分扑救、射正、门框。
- 先抓视觉形态明确的事件。
- 模型只看候选片段。
- 每个事件必须带视觉证据和置信度。
